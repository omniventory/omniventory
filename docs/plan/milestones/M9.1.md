> **Scope note (read first).** M9.1 is the **foundation** of the LLM milestone (M9). It ships **no user-facing AI feature** — only a place to **configure an OpenAI-compatible LLM provider** and **verify it works** before anything is built on top. It delivers, in **two phases** (A backend, B frontend) of **4 atomic steps**: a global LLM config (base URL / model / API key / enabled) stored in the existing `settings` key-value store with the API key as a **write-only secret** (no new table, **no migration**); an `LLMProvider` seam + an OpenAI-compatible `/v1/chat/completions` implementation (httpx, routed through the existing SSRF guard); a **staged "test connection"** flow (connectivity → model-answers → **multimodal probe**) modelled on the existing `/settings/email/test` diagnostic endpoint; and the admin **LLM settings section** on the Configuration page (reusing the existing `SecretField` + test-button patterns). The real AI features (receipt → batch entry, auto-categorize, semantic search) are **M9.2**, built additively on the seam this milestone reserves (§12).

## 1. Goal & non-goals

**Goal (the roadmap's M9.1 promise — a configurable, verifiable LLM base):**

- **Global LLM provider config** — one household-wide config (single-tenant, roadmap §1.2): `base_url`, `model`, `api_key` (secret), `enabled`. Stored in the existing **`settings`** KV store (the same store SMTP/HTTP/MQTT channel config uses), so there is **no new table and no migration**. The `api_key` is **write-only**: the API never echoes it, only a masked `api_key_is_set: bool` (the exact SMTP-password pattern).
- **`LLMProvider` seam + OpenAI-compatible implementation** — a `@runtime_checkable Protocol` in its own `app/services/llm/` package (the 4th instance of the repo's "Protocol seam + concrete impl + `build_*` factory + per-request DI" pattern, after product-lookup / search / notification-channels). The concrete provider speaks OpenAI **`/v1/chat/completions`** over `httpx`, copying the outbound discipline of the M4 HTTP channel (sync `httpx.Client`, `follow_redirects=False`, `raise_for_status`) and routing the URL through the **existing M6 SSRF guard** (`validate_outbound_url`).
- **Staged "test connection"** — an admin-only `POST /settings/llm/test` that **always returns HTTP 200** (a failed connection is a diagnostic outcome, not an API error — exactly the `/settings/email/test` contract) and reports a **three-stage capability check**: ① connectivity / auth, ② the configured model answers, ③ **multimodal probe** (send a small bundled fixture image and confirm the model reads it — receipt-scanning depends on this). Stages short-circuit; the result is purely diagnostic and never blocks saving/enabling.
- **Admin LLM settings section** — a new section on the Configuration page (`MANAGE_SETTINGS`-gated like the rest of it) with base-URL / model inputs, the reused `SecretField` for the API key, an enable switch, a Save action, and a **Test connection** button that renders the three staged results inline.

**Done when (🟢, expanded in §11):** as an admin, open **Configuration**, fill in the **base URL + API key + model** for an OpenAI-compatible provider (e.g. OpenRouter), **save** (the key is stored; reloading shows it as **set**, never as plaintext), hit **Test connection**, and watch the **three lights** report: connectivity ✓, model answers ✓, and **multimodal ✓** (the model read the bundled probe image). A bad key shows a clear **auth** failure on the connectivity light; a text-only model shows connectivity ✓ / model ✓ / **multimodal ✗** without blocking save. A non-admin cannot see the key, edit the config, or run the test. CI stays green, including the no-drift contract gate; **no migration is added** (head stays `0034`).

**Non-goals (explicitly out of M9.1 — deferred or M9.2):**
- **No AI features.** Receipt → batch product creation, product/barcode → auto-categorize, and semantic search are **M9.2**. M9.1 ships **no production code path that calls `chat()` for a user feature** — only the diagnostic test endpoint exercises the provider.
- **No streaming, no tool/function-calling, no structured-output mode, no token/cost accounting or budgets, no retry/backoff.** The provider does a single blocking request/response. These are noted refinements (§13); M9.2 adds what it needs.
- **No multiple providers / fallback chain.** One global config; the factory returns a **list-of-one** (mirroring `build_lookup_service`) so adding providers later is additive. No per-user LLM config.
- **No embedding / vector store.** Semantic search is M9.2 and leans toward **query expansion** over the existing `SearchProvider` LIKE seam, not a vector index (roadmap §5 / M9.2).
- **No localhost-loopback provider out of the box.** Outbound goes through the SSRF guard, which **blocks loopback** (and cloud-metadata) but **allows private LAN**. A LAN LLM (Ollama / LM Studio / vLLM on `192.168.x` / `10.x`) works; a same-host `127.0.0.1:11434` is blocked. A documented loopback allowance is deferred (§13). (OpenRouter and other remote HTTPS providers are unaffected.)
- **Not the OpenAI Responses API and not the Anthropic Messages format.** Chat Completions is the maximal-compatibility surface (roadmap §5, locked).
- **No model auto-discovery UI.** The model is typed in by hand; populating a dropdown from `GET /models` is a refinement (§13).

## 2. Locked decisions (settled in M9.1 planning; rationale in roadmap §5 M9)

| Area | Decision |
|---|---|
| **Config lives in the existing `settings` KV — no new table, no migration** | LLM config is user-facing, admin-editable runtime config, exactly like the M4 SMTP/HTTP/MQTT channels — so it belongs in the **`settings`** store (`SettingsService`), **not** the server-managed `app_config` store (which is walled off so secrets never reach the settings API). New dot-namespaced keys `llm.enabled` / `llm.base_url` / `llm.model` / `llm.api_key` are added to `SettingsService._DEFAULTS` (table stores overrides only). **Migration head stays `0034`.** |
| **API key is a *write-only* secret (the SMTP-password pattern)** | `llm.api_key` is registered in `SettingsService._SECRET_KEYS`. The read path emits **only** `api_key_is_set: bool` — the plaintext is **never** serialized to the API. The write path uses the established **omit = keep / `""` = clear / non-empty = set** convention (`if upd.api_key is not None: self._set_value("llm.api_key", upd.api_key if upd.api_key else None)`). The real key flows only internally, via an `LlmConfig` dataclass returned by a new `SettingsService.llm_config()` getter, consumed solely inside the provider. |
| **One global config; reads masked-open, writes admin-only** | Single household → one config (roadmap §1.2). The `llm` block rides the existing `GET/PATCH /settings`: `GET` is open to any authenticated user (safe — the key is masked); `PATCH` and the test endpoint require **`MANAGE_SETTINGS`** (admin) via the existing `require_manage_settings` dependency. No per-user config. |
| **API flavor: OpenAI `/v1/chat/completions` only** | The provider speaks **Chat Completions** — the broadest-supported surface across providers/gateways. **Not** the Responses API (many providers don't support it) and **not** Anthropic's Messages format. OpenRouter and similar OpenAI-compatible gateways work unchanged. Locked in roadmap §5. |
| **`LLMProvider` is its own seam in `app/services/llm/`** | A new package mirroring `app/services/product_lookup/`: `provider.py` (a `@runtime_checkable Protocol` + `@dataclass` request/result types), `openai.py` (the concrete impl, **duck-typed**, never inheriting the Protocol), `service.py` (`LLMService` + a `build_llm_service(db)` factory holding the provider list in one place). This is deliberately **distinct** from M5's two *reserved* LLM slots — "LLM vision" behind `ProductLookupProvider` and "LLM semantic search" behind `SearchProvider` — which are **higher-level consumers** that M9.2 will implement *on top of* this lower-level `LLMProvider`. Keeping it separate avoids colliding with those reserved names. |
| **Outbound copies the M4 HTTP channel + goes through the SSRF guard** | The OpenAI call mirrors `notifications/channels/http.py`: a per-call **sync** `httpx.Client(timeout=_TIMEOUT_SECONDS, follow_redirects=False)`, `Authorization: Bearer <key>`, `raise_for_status()`. The `base_url` is validated with **`app.core.net_guard.validate_outbound_url`** immediately before the request (lazy import, like the HTTP/MQTT call sites). The guard **blocks loopback + cloud-metadata** but **allows private LAN**, so a LAN LLM works and only same-host loopback is refused (→ §13). Timeout is a module constant set high for LLM latency (**`60.0`s**, vs the channel's `5.0`s). |
| **Test connection = staged, always-200 diagnostic (the `/settings/email/test` contract)** | `POST /settings/llm/test` is admin-only and **always returns HTTP 200** with a typed `LlmTestResult`; a failed connection is a *diagnostic outcome*, not an error envelope. It **ignores the `enabled` flag** (verify before enabling) and runs three stages that **short-circuit**: ① **connectivity/auth** (`GET {base_url}/models`; on 200 → pass, 401/403 → auth fail, connection/timeout/SSRF → connection fail); ② **model answers** (a minimal chat round-trip on the configured model; model-not-found → fail); ③ **multimodal** (a chat call carrying a **small bundled fixture image** as an OpenAI vision content-part, asking the model to read a known token; the token must appear in the reply). Each stage carries `{status: pass|fail|skipped, detail}`; later stages are `skipped` once an earlier one fails. |
| **Multimodal is *reported*, never *required*** | The three stages are diagnostic only — a failing multimodal light does **not** block saving or enabling. A text-only model (fine for, say, future semantic-expansion) saves cleanly; the multimodal-dependent feature (receipt scan, M9.2) is the one that checks the capability. Enabling is always allowed regardless of test outcome. |
| **Error codes (additions)** | New `llm.*` codes in `app/core/errors.py`: `llm.not_configured` (409 — operate with no base_url/model/api_key), `llm.unsafe_url` (422 — `base_url` blocked by the SSRF guard), `llm.connection_failed` (502 — unreachable / timeout / DNS), `llm.auth_failed` (502 — provider 401/403), `llm.model_unavailable` (502 — model not found/served), `llm.not_multimodal` (the multimodal-stage message), `llm.provider_error` (502 — any other provider error). In M9.1 these surface mainly as **test-stage details** + `llm.not_configured`; the rest are **reserved for M9.2 feature calls** (added now so the FE↔BE code contract is complete and localized from day one). All localized in the FE `errors` namespace (ZH+EN). |
| **Permissions (M6)** | `GET /settings` (incl. the masked `llm` block) → any authenticated user; `PATCH /settings` (incl. `llm`) and `POST /settings/llm/test` → `MANAGE_SETTINGS` (admin), via `require_manage_settings`. Writes audit-log through the existing `AuditService` like the other settings writes. |

> These extend M0–M6 foundations. **No new runtime dependency** — `httpx` is already a backend dep (M4 outbound webhook); the multimodal probe ships a tiny **static** PNG asset (no image library needed; `pillow` from M5 is available but unused here). The new endpoint + the `llm` settings block flow through the existing no-drift contract gate (§6) and the M1.5 uniform error envelope. `AGENTS.md` is **unchanged** — M9.1 is not a "foundations" touch.

## 3. Data model

**No schema change. No migration. Migration head stays `0034`.**

LLM config is stored as rows in the existing **`settings`** table (`key: String(128)` PK, `value: Text`, migration `0015`) via `SettingsService`, which already encodes booleans/strings and treats an empty string as the "cleared" sentinel. The four new keys (overrides only — defaults live in code):

| Setting key | Type | Default | Notes |
|---|---|---|---|
| `llm.enabled` | bool | `false` | Stored intent; consulted by M9.2 features + the provider factory's "is configured" check. The test endpoint **ignores** it. |
| `llm.base_url` | str \| None | `None` | The OpenAI-compatible API base (e.g. `https://openrouter.ai/api`). The provider appends `/v1/chat/completions` / `/models`. |
| `llm.model` | str \| None | `None` | The model id (e.g. `openai/gpt-4o-mini`). Typed by hand (no discovery UI — §13). |
| `llm.api_key` | str \| None (**secret**) | `None` | Registered in `_SECRET_KEYS`; **write-only**, surfaced only as `api_key_is_set`. |

The LLM provider writes **no** DB rows (no notifications, no audit beyond the settings-change audit the PATCH already emits). The bundled multimodal-probe image is a **static asset** committed under the `app/services/llm/` package (a few KB), not a DB blob.

## 4. Backend design

### 4.1 Layering (extends M4 channels / M5 provider seams / M6 settings)

- **New package `app/services/llm/`** (mirrors `app/services/product_lookup/`):
  - `provider.py` — `@runtime_checkable Protocol LLMProvider` with `chat(messages, *, model, max_tokens=None, temperature=None) -> ChatResult`; `@dataclass` types `ChatMessage(role, content)` (content is text **or** a list of parts so vision works), a small `image_part(...)` helper, and `ChatResult(text, model, raw)`. Documented contract: the provider **maps transport/provider failures to `AppError(llm.*)`** so callers (and the staged test) get stable codes.
  - `openai.py` — `OpenAICompatibleProvider(db)` reads `SettingsService(db).llm_config()`; builds the OpenAI Chat Completions payload; calls `validate_outbound_url(base_url)` then `httpx.Client(...).post(f"{base_url}/v1/chat/completions", json=…, headers={Authorization: Bearer …})`; maps results: connection/timeout/`UnsafeUrlError` → `llm.connection_failed`/`llm.unsafe_url`; 401/403 → `llm.auth_failed`; 404 / "model not found" → `llm.model_unavailable`; other non-2xx → `llm.provider_error`; parses the first choice's message content into `ChatResult`. A separate `list_models()` helper hits `GET {base_url}/models` for the connectivity probe.
  - `service.py` — `LLMService(db)` wrapping the provider; `test_connection() -> LlmTestResult` (the staged probe, §4.3); a `build_llm_service(db)` factory holding the provider list in one place (list-of-one now). `LLMService` also exposes `is_configured()` (base_url + api_key + model all set) for M9.2 + the FE.
  - **No raw queries** here; config access goes through `SettingsService` (roadmap §2.10).
- **`SettingsService` (extended, `app/services/settings.py`)**: add the four `llm.*` defaults to `_DEFAULTS`; add `llm.api_key` to `_SECRET_KEYS`; add `_build_llm_response()` (emits `api_key_is_set`), `_apply_llm_update()` (the omit/clear/set secret branch), and an `llm_config() -> LlmConfig` getter (mirrors `http_channel_config()` returning the real key for internal use only).
- **`app/core/errors.py` (extended)**: the seven `llm.*` `ErrorCode` constants under a new `# --- LLM (M9.1) ---` section + dev-facing `_DEFAULT_MESSAGES`.

### 4.2 Config schemas (`app/schemas/settings.py`)

- `LlmConfigResponse`: `enabled: bool`, `base_url: str | None`, `model: str | None`, `api_key_is_set: bool` (**no** `api_key` field) — mirrors `EmailChannelResponse`.
- `LlmConfigUpdate`: `enabled: bool | None = None`, `base_url: str | None = None`, `model: str | None = None`, `api_key: str | None = None` (`# write-only; "" = clear`) — mirrors `EmailChannelUpdate`.
- `SettingsResponse` gains `llm: LlmConfigResponse`; `SettingsUpdate` gains `llm: LlmConfigUpdate | None = None` (so the existing `GET/PATCH /settings` carries the block, inheriting its admin gate).
- The staged test result:
  - `LlmTestStage`: `status: Literal["pass","fail","skipped"]`, `detail: str | None`.
  - `LlmTestResult`: `ok: bool` (all non-skipped stages passed), `connectivity: LlmTestStage`, `model_answers: LlmTestStage`, `multimodal: LlmTestStage`.

### 4.3 The staged test (the easy-to-get-wrong logic — required tests)

`LLMService.test_connection()` runs three stages and short-circuits:

1. **connectivity** — `GET {base_url}/models` through the provider's `list_models()`. `200` → `pass`. `401/403` → `fail` (detail = `llm.auth_failed`). connection error / timeout / `UnsafeUrlError` → `fail` (`llm.connection_failed` / `llm.unsafe_url`). If config is incomplete (no base_url/key) → `fail` with `llm.not_configured`. (A provider that lacks `/models` but is otherwise reachable may surface a 404 here; the implementer may fall back to confirming reachability via the stage-2 chat ping — documented in the step note.)
2. **model_answers** — only if stage 1 passed; otherwise `skipped`. A minimal chat round-trip (`messages=[{role:user, content:"Reply with the single word: OK"}]`, small `max_tokens`) on the configured model. A parsed non-empty reply → `pass`. `llm.model_unavailable` → `fail`. other → `fail` (`llm.provider_error`).
3. **multimodal** — only if stage 2 passed; otherwise `skipped`. A chat call whose user message carries a **text part** ("What code is shown in this image? Reply with only the code.") **+ an image part** built from the bundled fixture (base64 `data:image/png;base64,…`). The fixture encodes a known short token; the stage **passes iff the reply contains that token** (case-insensitive). A provider/model that rejects images, or replies without the token → `fail` (`llm.not_multimodal`, detail carries the model's reply for the admin to judge).

`ok = connectivity.pass and model_answers.pass and multimodal.pass`. The route returns this verbatim with HTTP 200.

### 4.4 API surface (delta; all under `settings.api_prefix`, default `/api`)

| Method + path | Auth | Purpose |
|---|---|---|
| `GET /settings` | authenticated | now also carries the `llm` block (`enabled, base_url, model, api_key_is_set`). |
| `PATCH /settings` | `MANAGE_SETTINGS` | now also accepts `llm` (`enabled?, base_url?, model?, api_key?` with omit/clear/set on the key). Audit-logged. |
| `POST /settings/llm/test` | `MANAGE_SETTINGS` | run the staged probe; **always 200**; returns `LlmTestResult`. Ignores `enabled`. |

No other routes. The provider itself is internal (no public endpoint in M9.1 — features arrive in M9.2).

## 5. Quality gates & easy-to-get-wrong logic (Definition of Done)

Inherits M0–M6 §5 (`make check` all green; build; `make codegen` no-drift). The **logic that must have unit tests** in M9.1:

**Backend**
- **Secret masking (required):** the `llm` GET response carries **only** `api_key_is_set` and **never** the plaintext key (assert the serialized payload has no key value); `api_key_is_set` flips false→true after a PATCH that sets it; the **omit/clear/set** rule holds — omitting `api_key` keeps the stored value, `""` clears it (→ `api_key_is_set=false`), a non-empty value sets it; `base_url`/`model`/`enabled` round-trip.
- **Provider error mapping (required):** with the outbound `httpx` call **mocked** (no real network — anti-pollution), a `401/403` → `AppError(llm.auth_failed)`; a connection error / timeout → `llm.connection_failed`; a `base_url` that resolves to a blocked IP (loopback / `169.254.169.254`) → `llm.unsafe_url` (the SSRF guard fired); a `404`/model-not-found body → `llm.model_unavailable`; a generic 5xx → `llm.provider_error`; a well-formed 200 parses the first choice into `ChatResult`.
- **SSRF guard applied (required):** the provider calls `validate_outbound_url(base_url)` before the request; a LAN URL (`192.168.x` / `10.x`) is **allowed**, a loopback / metadata URL is **blocked** (parity with the HTTP channel).
- **Staged test logic (required):** all-pass yields `ok=true` and three `pass` stages; a connectivity failure makes stages 2 & 3 `skipped` and `ok=false`; a model failure makes stage 3 `skipped`; the multimodal stage **passes only when the reply contains the fixture token** and **fails** when the model omits it / rejects the image; incomplete config → connectivity `fail` with `llm.not_configured`; the endpoint **returns HTTP 200 in every case** (success and every failure).
- **Permissions (M6):** `viewer`/`member` blocked from `PATCH /settings` `llm` writes and from `POST /settings/llm/test` (403 `auth.forbidden`); admin permitted; the masked `GET` open to all authenticated.

**Frontend** (vitest + Testing Library, mock the typed client — M0 style): the **LLM section** loads the current config (key shown as **set/not-set**, never plaintext); editing base_url/model + typing a new key + Save sends the right PATCH body (**clear > new value > omit** on the key); **Test connection** drives `POST /settings/llm/test`, renders **three staged rows** (pass / fail / skipped) with the right colors, shows the multimodal light, and surfaces a failed connectivity light without blocking; server error **codes** localize via `mapApiError` (the `llm.*` family); all new strings present in **en + zh** (catalog-parity test).

## 9. Step breakdown (atomic, ordered)

Each step is independently testable, test-backed, lands exactly one commit (one per-step autosquash in orchestrator mode), re-runs `make codegen` if it touched the API, and inherits the global DoD (§5) + the **anti-freelancing rule**: implement *only* this step — no other steps, no opportunistic refactors, no touching the host's real environment (no real DB/containers/files/network outside the test sandbox; the provider is always **mocked** in tests).

> **Phasing:** A (backend: config → provider → test) → B (frontend). Backend steps are ordered 1 → 2 → 3 (2 reads the config from 1; 3 drives the provider from 2). The frontend step 4 consumes the contract from 1 + 3.

### Phase A — Backend

**Step 1 — LLM config in `settings` + schemas + error codes**
- **Build:** the four `llm.*` defaults in `_DEFAULTS`; `llm.api_key` in `_SECRET_KEYS`; `LlmConfig` dataclass + `llm_config()` getter; `_build_llm_response` / `_apply_llm_update`; `LlmConfigResponse` / `LlmConfigUpdate`; `llm` on `SettingsResponse` / `SettingsUpdate`; the seven `llm.*` `ErrorCode` constants + `_DEFAULT_MESSAGES`; `make codegen`.
- **Tests:** GET carries `api_key_is_set`, never the key; PATCH sets/keeps/clears the key (omit/`""`/value); base_url/model/enabled round-trip; viewer/member blocked from the `llm` PATCH (403 `auth.forbidden`); admin permitted.
- **Commit:** `feat(backend): LLM provider config in settings (write-only api key)`

**Step 2 — `LLMProvider` seam + OpenAI-compatible implementation**
- **Build:** `app/services/llm/` package — `provider.py` (Protocol + dataclasses), `openai.py` (httpx call copied from the HTTP channel, `validate_outbound_url`, 60s timeout, error mapping to `llm.*`, `list_models()`), `service.py` (`LLMService` + `is_configured()` + `build_llm_service(db)`). **No API change** (no `make codegen`).
- **Tests (required):** error mapping (401/403→auth, conn/timeout→connection, blocked IP→unsafe_url, 404/model→model_unavailable, 5xx→provider_error); happy 200 parses a `ChatResult`; SSRF guard invoked (LAN allowed, loopback/metadata blocked); **all outbound `httpx` mocked — no real network.**
- **Commit:** `feat(backend): OpenAI-compatible LLM provider seam`

**Step 3 — Staged "test connection" service + endpoint**
- **Build:** `LLMService.test_connection()` (connectivity → model → multimodal, short-circuit, bundled fixture image); `LlmTestStage` / `LlmTestResult` schemas; `POST /settings/llm/test` (admin-only, always 200); `make codegen`.

## 12. Reserved seam — what M9.2 hangs off (built here, consumed later)

> **M9.1 builds none of M9.2.** This records how the AI features attach to the base, so M9.2 is **additive, not a refactor**.

- **The `LLMProvider` seam** (`app/services/llm/`) is the single low-level entry point for every future LLM call. M9.2 features call `LLMService.chat(...)` (extended as needed with structured-output / vision helpers) — they never construct an httpx client or read the key themselves.
- **Receipt → batch product drafting (M9.2)** consumes `chat()` with the vision content-part (the same shape stage 3 proves works) to extract line items, then drafts `item_definitions` for review.
- **Auto-categorize (M9.2)** fills the **`ProductLookupProvider`** LLM slot M5 reserved — a new provider that *uses* `LLMService` internally and is appended to `build_lookup_service`'s list behind the `llm.enabled` toggle.
- **Semantic search (M9.2)** fills the **`SearchProvider`** slot M5 reserved — a query-expansion provider that *uses* `LLMService` to rewrite the query, then defers to the existing LIKE search (no vector store — roadmap §5).
- **Capability gating:** features check `LLMService.is_configured()` + `llm.enabled` (and, for receipt scan, the multimodal capability) before offering themselves; a clean `llm.not_configured` otherwise.

Nothing AI-feature-specific ships in M9.1; nothing in M9.2 requires reworking the M9.1 base.

## 13. Open questions / deferred

- **Localhost/loopback LLM:** the SSRF guard blocks `127.0.0.1`, so a same-host Ollama/LM Studio is refused while a LAN one works. A documented **allow-loopback toggle** (admin opt-in, narrowly scoped) is a contained later addition; deferred because the primary path is a remote gateway (OpenRouter) or a LAN server.
- **Model auto-discovery:** the model is typed by hand. Populating a dropdown from `GET /models` (the connectivity probe already hits it) is a UX refinement once the base is proven.
- **Streaming / function-calling / structured-output mode:** M9.1 does a single blocking request. M9.2 features that want JSON-mode / tool-calls / streaming add them on the provider as needed.
- **Token & cost accounting + budgets / rate-limiting:** not modelled. A usage ledger + per-period budget (and a guard before expensive calls) is a natural follow-up once real feature traffic exists.
- **Retry / backoff / circuit-breaking:** the provider does one attempt. Transient-failure retry policy is deferred to when feature traffic justifies it.
- **Multiple providers / fallback chain & per-user config:** one global provider (list-of-one factory). A fallback list or per-user keys would mirror the product-lookup multi-provider shape — deferred until a real need appears.
- **Secret encryption at rest:** the key is stored plaintext in the `settings` table (same as SMTP/MQTT secrets) — an operator with the DB file already has bigger problems (roadmap §5, locked). Encrypting channel/LLM secrets uniformly is a cross-cutting follow-up if ever wanted.
