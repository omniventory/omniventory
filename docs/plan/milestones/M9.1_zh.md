# M9.1 —— LLM 基座(provider 配置 + 测试连接)

> 🌐 **语言:** [English](./M9.1.md) · 中文(当前)

> **里程碑设计文档 —— 自包含。** 与 `docs/plan/roadmap.md`(地图,尤其 §5 **M9** + 记在那里的锁定决定,以及红线 **§2.10 单上下文 / 仓库层**、**§2.11 逻辑在 app 不在 SQL**、**§1.2 单租户 / 全体用户共享数据**、**§2.12 预留 provider 接缝**)一起读。本文是 *M9.1 建什么、怎么验* 的唯一真相源;别从 roadmap 反推范围。进度**只**在 roadmap §4 表里追踪。
>
> 已内建的约定:原子步(§9)、盲审 checkpoint(§10)、🟢 部署自测点(§11),让手动与编排两种执行方式都能挂在本文上。
>
> **范围提示(先读)。** M9.1 是 LLM 里程碑(M9)的**基座**。它**不交付任何面向用户的 AI 功能**——只交付一个能**配置 OpenAI 兼容 LLM provider** 并**验活**的地方,在往上建任何东西之前先打通。它分**两相**(A 后端、B 前端)、**4 个原子步**:一份全局 LLM 配置(base URL / model / API key / enabled),存进现有 `settings` 键值库、API key 作**只写密钥**(无新表、**无迁移**);一个 `LLMProvider` 接缝 + 一个 OpenAI 兼容 `/v1/chat/completions` 实现(httpx,过现有 SSRF 守卫);一个**分级"测试连接"**流程(连通 → 模型应答 → **多模态探活**),仿照现有 `/settings/email/test` 诊断端点;以及配置页上的管理员 **LLM 设置区块**(复用现成 `SecretField` + 测试按钮模式)。真正的 AI 功能(收据→批量录入、自动归类、语义检索)是 **M9.2**,加法式建在本里程碑预留的接缝上(§12)。

## 1. 目标与非目标

**目标(roadmap 的 M9.1 承诺 —— 一份可配置、可验证的 LLM 基座):**

- **全局 LLM provider 配置** —— 一份家庭级配置(单租户,roadmap §1.2):`base_url`、`model`、`api_key`(密钥)、`enabled`。存进现有 **`settings`** KV 库(SMTP/HTTP/MQTT 通道配置用的同一个库),所以**无新表、无迁移**。`api_key` **只写**:API 永不回传它,只返回遮罩后的 `api_key_is_set: bool`(完全是 SMTP 密码那套)。
- **`LLMProvider` 接缝 + OpenAI 兼容实现** —— 一个 `@runtime_checkable Protocol`,放在自己的 `app/services/llm/` 包里(仓库"Protocol 接缝 + 具体实现 + `build_*` 工厂 + per-request DI"模式的第 4 个实例,前三个是 product-lookup / search / notification-channels)。具体 provider 用 `httpx` 讲 OpenAI **`/v1/chat/completions`**,照搬 M4 HTTP 通道的出站纪律(同步 `httpx.Client`、`follow_redirects=False`、`raise_for_status`),并把 URL 过**现有 M6 SSRF 守卫**(`validate_outbound_url`)。
- **分级"测试连接"** —— 一个 admin-only 的 `POST /settings/llm/test`,**永远返回 HTTP 200**(连接失败是诊断结果、不是 API 错误 —— 正是 `/settings/email/test` 的契约),报告一个**三级能力检查**:① 连通 / 鉴权,② 配置的 model 能应答,③ **多模态探活**(发一张内置小测试图,核验模型能读图 —— 拍小票依赖它)。各级短路;结果纯诊断,绝不阻断保存/启用。
- **管理员 LLM 设置区块** —— 配置页上一个新区块(和其余部分一样受 `MANAGE_SETTINGS` 门控),含 base-URL / model 输入、复用的 `SecretField` 放 API key、一个启用开关、一个保存动作,以及一个把三级结果内联渲染的 **测试连接** 按钮。

**完成判据(🟢,§11 展开):** 以管理员身份打开 **配置**,为一个 OpenAI 兼容 provider(如 OpenRouter)填好 **base URL + API key + model**,**保存**(密钥被存下;重新加载显示为**已设置**,绝不显示明文),点 **测试连接**,看**三盏灯**报告:连通 ✓、模型应答 ✓、**多模态 ✓**(模型读出了内置探测图)。错误的 key 在连通灯上给出清楚的**鉴权**失败;纯文本模型给出 连通 ✓ / 模型 ✓ / **多模态 ✗** 但不阻断保存。非管理员看不到 key、改不了配置、跑不了测试。CI 保持绿,含 no-drift 契约门;**不新增迁移**(head 仍为 `0034`)。

**非目标(明确排除在 M9.1 外 —— 延后或归 M9.2):**
- **不做 AI 功能。** 收据→批量建档、商品/条码→自动归类、语义检索都是 **M9.2**。M9.1 **不交付任何为用户功能调用 `chat()` 的生产代码路径**——只有诊断测试端点会触达 provider。
- **不做流式、不做工具/函数调用、不做结构化输出模式、不做 token/成本计量或预算、不做重试/退避。** provider 做单次阻塞请求/响应。这些是已记的精化(§13);M9.2 按需补。
- **不做多 provider / 回退链。** 一份全局配置;工厂返回**单元素列表**(仿 `build_lookup_service`),将来加 provider 是加法。无 per-user LLM 配置。
- **不做 embedding / 向量库。** 语义检索是 M9.2,倾向在现有 `SearchProvider` LIKE 接缝上做**查询扩展**,而非向量索引(roadmap §5 / M9.2)。
- **开箱不支持 loopback 本机 provider。** 出站过 SSRF 守卫,它**挡 loopback**(以及云元数据)但**放行私网**。LAN 上的 LLM(`192.168.x` / `10.x` 的 Ollama / LM Studio / vLLM)可用;同机 `127.0.0.1:11434` 被挡。一个有文档的 loopback 放行开关延后(§13)。(OpenRouter 等远程 HTTPS provider 不受影响。)
- **不是 OpenAI Responses API,也不是 Anthropic Messages 格式。** Chat Completions 是最大兼容面(roadmap §5,锁定)。
- **不做模型自动发现 UI。** model 手动键入;从 `GET /models` 填下拉是精化(§13)。

## 2. 锁定决定(M9.1 规划期敲定;理由见 roadmap §5 M9)

| 方面 | 决定 |
|---|---|
| **配置进现有 `settings` KV —— 无新表、无迁移** | LLM 配置是面向用户、管理员可改的运行时配置,跟 M4 的 SMTP/HTTP/MQTT 通道完全一类 —— 所以归 **`settings`** 库(`SettingsService`),**不**进服务端自管的 `app_config` 库(后者被隔离,使密钥永不流入 settings API)。把点分命名键 `llm.enabled` / `llm.base_url` / `llm.model` / `llm.api_key` 加进 `SettingsService._DEFAULTS`(表只存覆盖值)。**迁移 head 仍为 `0034`。** |
| **API key 是*只写*密钥(SMTP 密码那套)** | `llm.api_key` 登记进 `SettingsService._SECRET_KEYS`。读路径**只**给 `api_key_is_set: bool`——明文**绝不**序列化到 API。写路径用既定的 **omit=保留 / `""`=清除 / 非空=设置** 约定(`if upd.api_key is not None: self._set_value("llm.api_key", upd.api_key if upd.api_key else None)`)。真实 key 只在内部流动,经新的 `SettingsService.llm_config()` getter 返回的 `LlmConfig` dataclass,仅 provider 内部消费。 |
| **一份全局配置;读遮罩开放、写 admin-only** | 单家庭 → 一份配置(roadmap §1.2)。`llm` 块搭现有 `GET/PATCH /settings`:`GET` 对任何已认证用户开放(安全——key 已遮罩);`PATCH` 与测试端点要 **`MANAGE_SETTINGS`**(管理员),经现有 `require_manage_settings` 依赖。无 per-user 配置。 |
| **API 风味:只用 OpenAI `/v1/chat/completions`** | provider 讲 **Chat Completions** —— 跨 provider/网关支持最广的面。**不用** Responses API(很多 provider 不支持),**不用** Anthropic Messages 格式。OpenRouter 等 OpenAI 兼容网关原样可用。roadmap §5 锁定。 |
| **`LLMProvider` 是 `app/services/llm/` 里自己的接缝** | 一个仿 `app/services/product_lookup/` 的新包:`provider.py`(一个 `@runtime_checkable Protocol` + `@dataclass` 请求/结果类型)、`openai.py`(具体实现,**鸭子类型**,从不继承 Protocol)、`service.py`(`LLMService` + `build_llm_service(db)` 工厂,把 provider 列表收在一处)。这刻意**区别于** M5 那两个*预留的* LLM 槽 —— `ProductLookupProvider` 背后的"LLM vision"和 `SearchProvider` 背后的"LLM 语义搜索"—— 它们是 M9.2 将在这个更底层的 `LLMProvider` *之上*实现的**更高层消费者**。分开放避免和那些预留名撞车。 |
| **出站照搬 M4 HTTP 通道 + 过 SSRF 守卫** | OpenAI 调用镜像 `notifications/channels/http.py`:每次调用一个**同步** `httpx.Client(timeout=_TIMEOUT_SECONDS, follow_redirects=False)`、`Authorization: Bearer <key>`、`raise_for_status()`。`base_url` 在请求前用 **`app.core.net_guard.validate_outbound_url`** 校验(惰性导入,和 HTTP/MQTT 调用点一致)。守卫**挡 loopback + 云元数据**但**放行私网**,所以 LAN LLM 可用、只有同机 loopback 被拒(→ §13)。超时是个为 LLM 延迟调高的模块常量(**`60.0`s**,对比通道的 `5.0`s)。 |
| **测试连接 = 分级、永远 200 的诊断(`/settings/email/test` 契约)** | `POST /settings/llm/test` admin-only、**永远返回 HTTP 200**,带一个有类型的 `LlmTestResult`;连接失败是*诊断结果*,不是错误信封。它**忽略 `enabled` 标志**(先验后启用),跑三级、**短路**:① **连通/鉴权**(`GET {base_url}/models`;200 → pass,401/403 → 鉴权失败,连接/超时/SSRF → 连通失败);② **模型应答**(在配置 model 上做最小 chat 往返;model 不存在 → 失败);③ **多模态**(一个 chat 调用,带一张**内置小测试图**作 OpenAI vision content-part,让模型读出已知 token;token 必须出现在回复里)。每级带 `{status: pass|fail|skipped, detail}`;一旦前级失败,后级 `skipped`。 |
| **多模态是*报告*,绝不*强制*** | 三级纯诊断 —— 多模态灯失败**不**阻断保存或启用。纯文本模型(对将来的语义扩展等就够用)照样干净保存;依赖多模态的功能(拍小票,M9.2)才去检查该能力。无论测试结果如何,启用始终允许。 |
| **错误码(新增)** | `app/core/errors.py` 新增 `llm.*` 码:`llm.not_configured`(409 —— 在无 base_url/model/api_key 时操作)、`llm.unsafe_url`(422 —— `base_url` 被 SSRF 守卫挡)、`llm.connection_failed`(502 —— 不可达/超时/DNS)、`llm.auth_failed`(502 —— provider 401/403)、`llm.model_unavailable`(502 —— model 不存在/未提供)、`llm.not_multimodal`(多模态级的消息)、`llm.provider_error`(502 —— 其它 provider 错误)。M9.1 里它们主要作**测试级 detail** + `llm.not_configured` 出现;其余**为 M9.2 功能调用预留**(现在就加,使 FE↔BE 码契约从第一天起完整且本地化)。全部在 FE `errors` 命名空间本地化(ZH+EN)。 |
| **权限(M6)** | `GET /settings`(含遮罩后的 `llm` 块)→ 任何已认证用户;`PATCH /settings`(含 `llm`)与 `POST /settings/llm/test` → `MANAGE_SETTINGS`(管理员),经 `require_manage_settings`。写入像其它 settings 写入一样经现有 `AuditService` 审计。 |

> 这些扩展 M0–M6 基础。**无新运行时依赖** —— `httpx` 已是后端依赖(M4 出站 webhook);多模态探测图随包带一张极小**静态** PNG(不需图像库;M5 来的 `pillow` 可用但此处不用)。新端点 + `llm` 设置块走现有 no-drift 契约门(§6)与 M1.5 统一错误信封。`AGENTS.md` **不变** —— M9.1 不是"基础设施"改动。

## 3. 数据模型

**无 schema 改动。无迁移。迁移 head 仍为 `0034`。**

LLM 配置作为现有 **`settings`** 表(`key: String(128)` PK、`value: Text`、迁移 `0015`)里的行,经 `SettingsService` 存储——后者已能编码布尔/字符串、并把空串当"已清除"哨兵。四个新键(只存覆盖值——默认在代码里):

| 设置键 | 类型 | 默认 | 说明 |
|---|---|---|---|
| `llm.enabled` | bool | `false` | 存的意图;由 M9.2 功能 + provider 工厂的"是否已配置"检查读取。测试端点**忽略**它。 |
| `llm.base_url` | str \| None | `None` | OpenAI 兼容 API 基址(如 `https://openrouter.ai/api`)。provider 追加 `/v1/chat/completions` / `/models`。 |
| `llm.model` | str \| None | `None` | model id(如 `openai/gpt-4o-mini`)。手动键入(无发现 UI —— §13)。 |
| `llm.api_key` | str \| None(**密钥**) | `None` | 登记进 `_SECRET_KEYS`;**只写**,仅以 `api_key_is_set` 呈现。 |

LLM provider **不**写任何 DB 行(无通知、除 PATCH 已发的 settings-change 审计外无额外审计)。多模态探测图是 `app/services/llm/` 包下一份**静态资源**(几 KB),不是 DB blob。

## 4. 后端设计

### 4.1 分层(扩展 M4 通道 / M5 provider 接缝 / M6 设置)

- **新包 `app/services/llm/`**(仿 `app/services/product_lookup/`):
  - `provider.py` —— `@runtime_checkable Protocol LLMProvider`,带 `chat(messages, *, model, max_tokens=None, temperature=None) -> ChatResult`;`@dataclass` 类型 `ChatMessage(role, content)`(content 为文本**或**部件列表,使 vision 可用)、一个 `image_part(...)` 助手、`ChatResult(text, model, raw)`。约定:provider **把传输/provider 失败映射成 `AppError(llm.*)`**,使调用方(与分级测试)拿到稳定码。
  - `openai.py` —— `OpenAICompatibleProvider(db)` 读 `SettingsService(db).llm_config()`;构造 OpenAI Chat Completions 载荷;调 `validate_outbound_url(base_url)` 后 `httpx.Client(...).post(f"{base_url}/v1/chat/completions", json=…, headers={Authorization: Bearer …})`;映射:连接/超时/`UnsafeUrlError` → `llm.connection_failed`/`llm.unsafe_url`;401/403 → `llm.auth_failed`;404 / "model not found" → `llm.model_unavailable`;其它非 2xx → `llm.provider_error`;把第一个 choice 的 message content 解析成 `ChatResult`。另有 `list_models()` 助手打 `GET {base_url}/models` 供连通探活。
  - `service.py` —— `LLMService(db)` 包住 provider;`test_connection() -> LlmTestResult`(分级探活,§4.3);`build_llm_service(db)` 工厂把 provider 列表收在一处(现在单元素)。`LLMService` 还暴露 `is_configured()`(base_url + api_key + model 都已设)供 M9.2 + 前端。
  - 此处**无裸查询**;配置访问走 `SettingsService`(roadmap §2.10)。
- **`SettingsService`(扩展,`app/services/settings.py`)**:把四个 `llm.*` 默认加进 `_DEFAULTS`;把 `llm.api_key` 加进 `_SECRET_KEYS`;加 `_build_llm_response()`(给 `api_key_is_set`)、`_apply_llm_update()`(omit/clear/set 密钥分支)、一个 `llm_config() -> LlmConfig` getter(仿 `http_channel_config()`,返回真实 key 仅供内部用)。
- **`app/core/errors.py`(扩展)**:在新 `# --- LLM (M9.1) ---` 段下加七个 `llm.*` `ErrorCode` 常量 + dev 用 `_DEFAULT_MESSAGES`。

### 4.2 配置 schema(`app/schemas/settings.py`)

- `LlmConfigResponse`:`enabled: bool`、`base_url: str | None`、`model: str | None`、`api_key_is_set: bool`(**无** `api_key` 字段)—— 仿 `EmailChannelResponse`。
- `LlmConfigUpdate`:`enabled: bool | None = None`、`base_url: str | None = None`、`model: str | None = None`、`api_key: str | None = None`(`# 只写;"" = 清除`)—— 仿 `EmailChannelUpdate`。
- `SettingsResponse` 加 `llm: LlmConfigResponse`;`SettingsUpdate` 加 `llm: LlmConfigUpdate | None = None`(使现有 `GET/PATCH /settings` 带上该块,继承其 admin 门)。
- 分级测试结果:
  - `LlmTestStage`:`status: Literal["pass","fail","skipped"]`、`detail: str | None`。
  - `LlmTestResult`:`ok: bool`(所有非 skipped 级都通过)、`connectivity: LlmTestStage`、`model_answers: LlmTestStage`、`multimodal: LlmTestStage`。

### 4.3 分级测试(易错逻辑 —— 必测)

`LLMService.test_connection()` 跑三级并短路:

1. **connectivity** —— 经 provider 的 `list_models()` 打 `GET {base_url}/models`。`200` → `pass`。`401/403` → `fail`(detail = `llm.auth_failed`)。连接错/超时/`UnsafeUrlError` → `fail`(`llm.connection_failed` / `llm.unsafe_url`)。配置不全(无 base_url/key)→ `fail`,`llm.not_configured`。(一个可达但无 `/models` 的 provider 可能在此回 404;实现者可退而用第 2 级 chat ping 确认可达——在步骤说明里记。)
2. **model_answers** —— 仅当第 1 级 pass;否则 `skipped`。在配置 model 上做最小 chat 往返(`messages=[{role:user, content:"Reply with the single word: OK"}]`,小 `max_tokens`)。解析出非空回复 → `pass`。`llm.model_unavailable` → `fail`。其它 → `fail`(`llm.provider_error`)。
3. **multimodal** —— 仅当第 2 级 pass;否则 `skipped`。一个 chat 调用,其 user 消息带一个**文本部件**("What code is shown in this image? Reply with only the code.")**+ 一个图像部件**,由内置 fixture 构造(base64 `data:image/png;base64,…`)。fixture 编码一个已知短 token;该级**当且仅当回复包含该 token(忽略大小写)时 pass**。拒收图像的 provider/model,或回复里没那 token → `fail`(`llm.not_multimodal`,detail 带模型回复供管理员判断)。

`ok = connectivity.pass and model_answers.pass and multimodal.pass`。路由原样以 HTTP 200 返回。

### 4.4 API 面(增量;均在 `settings.api_prefix` 下,默认 `/api`)

| 方法 + 路径 | 鉴权 | 用途 |
|---|---|---|
| `GET /settings` | 已认证 | 现在也带 `llm` 块(`enabled, base_url, model, api_key_is_set`)。 |
| `PATCH /settings` | `MANAGE_SETTINGS` | 现在也接受 `llm`(`enabled?, base_url?, model?, api_key?`,key 走 omit/clear/set)。审计。 |
| `POST /settings/llm/test` | `MANAGE_SETTINGS` | 跑分级探活;**永远 200**;返回 `LlmTestResult`。忽略 `enabled`。 |

无其它路由。provider 本身是内部的(M9.1 无公开端点 —— 功能 M9.2 到)。

## 5. 质量门与易错逻辑(完成定义)

继承 M0–M6 §5(`make check` 全绿;构建;`make codegen` no-drift)。M9.1 里**必须有单测的逻辑**:

**后端**
- **密钥遮罩(必测):** `llm` GET 响应**只**带 `api_key_is_set`、**绝不**带明文 key(断言序列化载荷里没有 key 值);PATCH 设置后 `api_key_is_set` 由 false→true;**omit/clear/set** 规则成立 —— 省略 `api_key` 保留原值、`""` 清除(→ `api_key_is_set=false`)、非空设置;`base_url`/`model`/`enabled` 往返。
- **provider 错误映射(必测):** 把出站 `httpx` 调用**打桩**(无真实网络——防污染),`401/403` → `AppError(llm.auth_failed)`;连接错/超时 → `llm.connection_failed`;一个解析到被挡 IP(loopback / `169.254.169.254`)的 `base_url` → `llm.unsafe_url`(SSRF 守卫触发);`404`/model-not-found 体 → `llm.model_unavailable`;通用 5xx → `llm.provider_error`;一个良构 200 把第一个 choice 解析成 `ChatResult`。
- **SSRF 守卫已应用(必测):** provider 在请求前调 `validate_outbound_url(base_url)`;LAN URL(`192.168.x` / `10.x`)**放行**,loopback / 元数据 URL **被挡**(与 HTTP 通道一致)。
- **分级测试逻辑(必测):** 全通 → `ok=true` 且三级 `pass`;连通失败使第 2、3 级 `skipped` 且 `ok=false`;模型失败使第 3 级 `skipped`;多模态级**仅当回复含 fixture token 时 pass**、模型漏掉/拒图时 **fail**;配置不全 → 连通 `fail` 带 `llm.not_configured`;端点在**每种情形下都返回 HTTP 200**(成功与每种失败)。
- **权限(M6):** `viewer`/`member` 被挡在 `PATCH /settings` 的 `llm` 写与 `POST /settings/llm/test` 外(403 `auth.forbidden`);admin 允许;遮罩 `GET` 对所有已认证开放。

**前端**(vitest + Testing Library,打桩有类型客户端 —— M0 风格):**LLM 区块**加载当前配置(key 显示**已设置/未设置**、绝不明文);改 base_url/model + 键入新 key + 保存,发出正确 PATCH 体(key 走 **clear > new > omit**);**测试连接** 驱动 `POST /settings/llm/test`,渲染**三个分级行**(pass / fail / skipped)及对应颜色,显示多模态灯,且连通灯失败时不阻断;服务端错误**码**经 `mapApiError` 本地化(`llm.*` 族);所有新串 **en + zh** 齐备(目录-parity 测试)。

## 9. 步骤拆解(原子、有序)

每步独立可测、有测试背书、恰好落一个提交(编排模式下一次 per-step autosquash)、若触 API 则重跑 `make codegen`,并继承全局 DoD(§5)+ **反夹带规则**:只实现*本步* —— 不做别的步、不顺手重构、不碰宿主真实环境(测试沙箱外无真实 DB/容器/文件/网络;测试里 provider 始终**打桩**)。

> **分相:** A(后端:配置 → provider → 测试)→ B(前端)。后端步按 1 → 2 → 3(2 读 1 的配置;3 驱动 2 的 provider)。前端第 4 步消费 1 + 3 的契约。

### A 相 —— 后端

## 12. 预留接缝 —— M9.2 挂在什么上(本里程碑建、后续消费)

> **M9.1 不建 M9.2 的任何东西。** 这里记 AI 功能怎么接到基座上,使 M9.2 是**加法、非重构**。

- **`LLMProvider` 接缝**(`app/services/llm/`)是将来每一次 LLM 调用的唯一底层入口。M9.2 功能调 `LLMService.chat(...)`(按需用结构化输出 / vision 助手扩展)—— 它们从不自己构造 httpx 客户端或读 key。
- **收据 → 批量起草商品(M9.2)** 用带 vision content-part 的 `chat()`(就是第 3 级证明可用的那个形状)抽行项,再草拟 `item_definitions` 待审。
- **自动归类(M9.2)** 补上 M5 预留的 **`ProductLookupProvider`** LLM 槽 —— 一个内部*使用* `LLMService`、在 `llm.enabled` 开关后追加进 `build_lookup_service` 列表的新 provider。
- **语义检索(M9.2)** 补上 M5 预留的 **`SearchProvider`** 槽 —— 一个*使用* `LLMService` 改写查询、再交给现有 LIKE 搜索的查询扩展 provider(无向量库 —— roadmap §5)。
- **能力门控:** 功能在自我提供前检查 `LLMService.is_configured()` + `llm.enabled`(拍小票还查多模态能力);否则给一个干净的 `llm.not_configured`。

M9.1 不出任何 AI 功能专属代码;M9.2 也不需要返工 M9.1 基座。

## 13. 待定 / 推迟

- **本机/loopback LLM:** SSRF 守卫挡 `127.0.0.1`,故同机 Ollama/LM Studio 被拒、而 LAN 上的可用。一个有文档的**允许 loopback 开关**(管理员显式开、范围窄)是可控的后加项;之所以推迟,是因为主路径是远程网关(OpenRouter)或 LAN 服务器。
- **模型自动发现:** model 手动键入。从 `GET /models`(连通探活已在打它)填下拉是基座验稳后的 UX 精化。
- **流式 / 函数调用 / 结构化输出模式:** M9.1 做单次阻塞请求。M9.2 里想要 JSON 模式 / 工具调用 / 流式的功能,按需在 provider 上加。
- **token & 成本计量 + 预算 / 限流:** 未建模。一个用量账本 + 周期预算(以及昂贵调用前的守卫)是真实功能流量出现后的自然后续。
- **重试 / 退避 / 熔断:** provider 单次尝试。瞬时失败重试策略推迟到功能流量值得时。
- **多 provider / 回退链 & per-user 配置:** 一份全局 provider(单元素列表工厂)。回退列表或 per-user key 会镜像 product-lookup 的多 provider 形状 —— 待真实需求出现再做。
- **密钥静态加密:** key 明文存在 `settings` 表(与 SMTP/MQTT 密钥一样)—— 能拿到 DB 文件的运维者本就有更大麻烦(roadmap §5,锁定)。统一加密通道/LLM 密钥是若需要时的横切后续。
