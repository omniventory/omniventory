## 1. Goal & non-goals

**Goal (the roadmap's M0 promise):** an *empty-but-running*, CI-green app you can log into. No domain features yet — just the skeleton and the quality gates that every later milestone leans on.

**Done when (🟢, expanded in §11):** app boots; `GET /api/health` is green; login works; one Alembic migration applies cleanly to an empty DB; CI is green; `docker build` produces a single image that runs the whole thing.

**Non-goals (explicitly out of M0):**
- No domain model beyond the `Household/Workspace` singleton — **no** Item/Definition/Lot/Location/Category/Movement tables yet (those start at M1).
- No multi-user, roles, invitations (M6). M0 creates **exactly one admin** user via a minimal first-run onboarding page. No general registration UI.
- No reminders, no email/SMTP, no barcode, no CSV/IO, no LLM hooks.
- No production hardening beyond baseline hygiene (security hardening is M6).
- No bespoke visual design — the frontend gets a **lean theme foundation + responsive shell only** (§7).

## 2. Locked decisions (settled in planning; rationale in chat history & roadmap §1)

| Area | Decision |
|---|---|
| Backend lang/tooling | **Python 3.13**, managed by **`uv`**; lint/format **ruff**, type-check **mypy**, tests **pytest**. |
| Web framework | **FastAPI** + **Pydantic v2** (`pydantic-settings` for config). |
| API surface | **No URL version prefix** in v1 (`/api/...`); all routers mounted under one configurable `api_prefix`, plus an integer **`api_version`** exposed in `/api/health` for compatibility detection. Add `/api/v2` only if M9's public API ever needs a breaking change. |
| Persistence | **SQLAlchemy 2.0** (typed, `Mapped[...]`) + **Alembic** migrations + **SQLite** (default, file on a volume). |
| Frontend lang/tooling | **React + TypeScript** + **Vite**, package manager **`pnpm`**; lint **eslint**, type-check **tsc**, tests **vitest** (+ Testing Library). |
| UI | **Mantine** component lib + **`react-feather`** icons; responsive (desktop & mobile equally first-class); **PWA** (installable, offline app-shell). |
| Auth | **Session-cookie**: opaque session id in an `HttpOnly` + `Secure` + `SameSite=Lax` cookie; server-side `sessions` table in SQLite. No JWT in v1. `secret_key` is **auto-generated & persisted** in `app_config` on first run if not set via env. First-run **onboarding** (`GET/POST /auth/setup-status` / `/auth/setup`) replaces the env-based admin bootstrap. |
| Contract | **Contract-first**: FastAPI OpenAPI → `openapi-typescript` → `frontend/src/api/schema.d.ts`; runtime calls via **`openapi-fetch`**. **No-drift CI gate** (§6). |
| Repo shape | **Monorepo**: this repo, root `backend/` + `frontend/`. |
| Deploy | **Single Docker container**: embedded built frontend + FastAPI + SQLite (bind mount). **Prod `docker-compose.yml`** (prebuilt image, no `build`) + **`docker-compose.dev.yml`** override (adds `build` for dev). Migrations run in a dedicated one-shot `migrate` service; the app starts only after it succeeds (fail-closed). |

## 4. Backend skeleton

### 4.1 App & config
- **App factory** `create_app()` in `app/main.py`: builds the FastAPI instance, mounts **all routers under one configurable `settings.api_prefix`** (default `/api`) — so adding a `/api/v1` (or `/v2`) prefix later is a one-line change — and wires middleware (session, CORS only if needed in dev). Keeps import-time side effects out.
- **Settings** via `pydantic-settings` (`app/config.py`): `api_prefix` (default `/api`), `api_version` (compatibility number, see §4.2), `database_url` (default `sqlite:///./data/omniventory.db`), `secret_key` (**optional** — auto-generated & persisted in `app_config` on first run if unset), `session_cookie_name`, `environment`. Loaded from env / `.env`. **No hardcoded secrets.**

### 4.2 Health
- `GET {api_prefix}/health` → `{ status: "ok", version, api_version, db: "ok" }`. The `db` field does a trivial `SELECT 1` so health reflects DB reachability. **`api_version`** is an integer compatibility number (InvenTree-style) — our chosen alternative to URL versioning: clients/tools detect compatibility from it without us versioning the URL. This is a 🟢 checkpoint.

### 4.3 Persistence & migrations
- **SQLAlchemy 2.0** typed models on a single `DeclarativeBase`. Engine + `sessionmaker` in `app/db/base.py`.
- **Alembic** wired against the same metadata; `env.py` reads `database_url` from Settings. **First migration** creates `households`, `users`, `sessions`.
- **Business logic stays in the service layer** — no SQLite-specific SQL, views, or triggers (keeps the DB swappable; roadmap constraint §2.11/§2.1).

### 4.4 Data-access / context layer (the multi-tenant hedge — roadmap §1.2)
- All DB access goes through a **repository layer** (`app/repositories/`) + a single **"current context"** abstraction (`app/core/context.py`) — never scattered raw queries in route handlers.
- The context carries the *current household* + *current user*; today it resolves to the singleton household and the logged-in user. This is the seam that makes a future `household_id` scope filter a contained change, not a rewrite.

### 4.5 `Household/Workspace` singleton
- One row: `name`, `currency`, `timezone`, `settings` (JSON). Created by the first migration (or first-run bootstrap). A small invariant guard ensures it stays a singleton in M0.

### 4.6 Session-cookie auth skeleton
- **Password hashing**: `argon2` (via `argon2-cffi` / passlib) — never store plaintext.
- **`sessions` table**: `id` (opaque random, e.g. `secrets.token_urlsafe`), `user_id`, `created_at`, `expires_at`, `last_seen_at`. Sliding or fixed expiry (fixed is fine for M0).
- **Endpoints** (`app/api/routes/auth.py`): `POST /api/auth/login` (sets cookie), `POST /api/auth/logout` (deletes server-side session + clears cookie), `GET /api/auth/me` (current user; 401 if no/invalid session).
- **First-run onboarding endpoints** (unauthenticated): `GET /api/auth/setup-status` → `{ setup_required: bool }` (true when zero users exist); `POST /api/auth/setup` → body `{ email, password }` — creates the first admin; **409 Conflict** if any user already exists (self-closing). Does NOT auto-login.
- **Secret key resolution** (at lifespan startup): if `SECRET_KEY` env is set and non-empty → use it; else read `app_config['secret_key']`; else generate `secrets.token_hex(32)`, persist to `app_config`, use it. Stashed on `app.state.secret_key`. Env value is never persisted.
- **Cookie**: `HttpOnly`, `Secure`, `SameSite=Lax`; value = opaque session id (no user data in it). Revocation = delete the row.
- **Dependency** `current_user` in `app/api/deps.py` reads the cookie, looks up + validates the session, yields the user (or 401).

## 6. Contract-first codegen & no-drift gate

Flow:
1. `backend/scripts/export_openapi.py` imports the app and writes the OpenAPI document to repo-root **`openapi.json`**.
2. `openapi-typescript openapi.json -o frontend/src/api/schema.d.ts` regenerates the **type-only** declarations.
3. `frontend/src/api/client.ts` wraps `openapi-fetch` with `schema.d.ts` so every call is path/param/response-typed.
4. **Both `openapi.json` and `schema.d.ts` are committed.**

**No-drift CI gate:** CI runs `make codegen` (= steps 1–2), then `git diff --exit-code openapi.json frontend/src/api/schema.d.ts`. If the backend API changed but the committed artifacts weren't regenerated, the diff is non-empty → **CI fails**. This is the mechanism the author asked for ("if there's drift it won't pass").

## 7. Frontend skeleton & style foundation (lean)

### 7.1 Toolchain
- Vite + React + TS; `pnpm` scripts: `dev`, `build`, `lint`, `typecheck`, `test`, `codegen`. (Run scripts with `pnpm <script>`.)

### 7.2 Mantine theme foundation — the "style foundation" decision (lean customization)
Centralized in `src/theme.ts`, so a future re-skin = editing this one object:
- **primary color**: a calm, "trustworthy inventory" hue — proposal **`teal`** (or `indigo`); one-line change.
- **font / radius / spacing**: Mantine defaults (no fight with the lib).
- **dark mode: day-one** — follow system + manual toggle, wired into the shell from the start (cheap now, annoying to retrofit).
- Wrapped by `MantineProvider` + color-scheme management at the app root.

### 7.3 Responsive app shell
- Mantine **`AppShell`**: desktop = persistent **sidebar (navbar) + header**; mobile = collapsed into a **burger → `Drawer`** (bottom-tab-bar is an open question, §12 — default to drawer for M0). **One** shell definition; every later page mounts inside it.

### 7.4 PWA shell
- `vite-plugin-pwa`: web manifest (name, icons, theme color, `display: standalone`), service worker precaching the **app shell** for offline-installable behavior. (Offline = static assets only; data/auth is online.)

### 7.5 Seed shared patterns (keep tiny)
- `components/`: `PageShell`, `LoadingState`, `EmptyState`, `ErrorState`. Just enough so M1+ pages don't each reinvent these. **Do not** build a component library here.

### 7.6 Auth UI
- **Gate logic in `App.tsx`**: on load, first call `GET /api/auth/setup-status`; if `setup_required` → show Setup; else call `GET /api/auth/me` → shell (authed) or Login (anon).
- `pages/Setup.tsx` — first-run form; on success → transitions to Login (no auto-login).
- `pages/Login.tsx` posts to `/api/auth/login` via the typed client; on success routes into the shell. A route guard calls `/api/auth/me`; unauthenticated → login. Header has a logout action.

> A standalone bilingual *design-system* doc is **deferred** (§12) until the UI grows; M0 captures the foundation here.

## 8. CI & Docker

### 8.1 CI (`.github/workflows/ci.yml`)
Jobs (parallel where possible), all must be green:
- **backend**: setup uv (+ cache), `ruff`, `mypy`, `pytest`, alembic upgrade against a temp SQLite (migration applies cleanly).
- **frontend**: setup pnpm (**+ pnpm store cache**) & node, `eslint`, `tsc`, `vitest`, `vite build`.
- **contract**: `make codegen` + `git diff --exit-code` (the no-drift gate).
- **docker**: `docker build` the single image; `docker compose config` validates the compose files; smoke test runs the one-shot **migration** then starts the container and checks `/api/health` (`db:ok`) + `/api/auth/setup-status`.

> CI minutes matter (free-tier cap): **cache the pnpm store and the uv cache** so installs are near-instant on repeat runs.

### 8.2 Docker (`docker/Dockerfile`, multi-stage → single image)
1. **stage 1 (frontend)**: `pnpm install --frozen-lockfile` + `pnpm build` → static assets.
2. **stage 2 (backend)**: `uv sync --frozen` into a slim Python base.
3. **final**: copy backend + built frontend; FastAPI serves the API under `/api/*` and the static SPA for everything else; SQLite on a **bind mount** (`/app/data`; no `VOLUME` directive); **the entrypoint does NOT run migrations** — a dedicated compose `migrate` service does (§8.3); `HEALTHCHECK` hits `/api/health`. Single container, one port. Container runs as **uid/gid `1000:1000`** so the bind-mounted SQLite file is owned by the host user — easy to back up without sudo.

### 8.3 Compose (production + dev override)
Two root-level compose files, layered via Compose's override mechanism:
- **`docker-compose.yml` (production)** defines two services:
  - **`migrate`** — a one-shot runner (`command: ["alembic", "upgrade", "head"]`) on the same image / bind mount / `user`, `restart: "no"`, healthcheck disabled. It applies migrations and exits.
  - **`app`** — runs from a **prebuilt `image:`** (no `build:`), SQLite on the same **bind mount** (`${DATA_DIR:-./data}:/app/data`; no named volume), `user: "1000:1000"` for host-owned files, host port from `${APP_PORT:-8000}`, env from `.env`, `restart`, healthcheck. It **`depends_on` `migrate` with `condition: service_completed_successfully`**, so the app starts **only after migrations succeed**; if a migration fails the app is never started (**fail-closed**).
- **`docker-compose.dev.yml` (dev override)**: adds **`build:`** (context `.`, `docker/Dockerfile`) + dev env to **both** services (shared via a YAML anchor); designed to be stacked on top of the prod file.
- **Dev run command:**
  ```bash
  docker compose -f docker-compose.yml -f docker-compose.dev.yml up -d --build
  ```
  Production runs from the prod file alone (`docker compose up -d`, uses the prebuilt image, no build).
- **`.env.example`** documents `APP_PORT`, `DATA_DIR`, and `SECRET_KEY` (blank = auto-generated).

## 9. Step breakdown (atomic, ordered)

Each step is independently testable, test-backed, and lands exactly one commit (one per-step autosquash in orchestrator mode). Steps are written for **blind orchestration** — a fresh implementer should be able to build the step from this section + the doc alone. Every step inherits the global Definition of Done (§5) and the **anti-freelancing rule**: implement *only* this step — no other steps, no opportunistic refactors, no touching the host's real environment.

### Step 1 — Monorepo scaffolding, toolchains & CI skeleton
- **Goal:** the repo laid out per §3, both toolchains installed, every gate green on a trivial sample.
- **Build:**
  - Root: `Makefile` (`check`/`lint`/`test`/`codegen` aliases; `codegen` may be a stub until step 5), `.gitignore`, `.dockerignore`, `.github/workflows/ci.yml`.
  - `backend/`: `pyproject.toml` (uv project; runtime deps `fastapi`, `uvicorn[standard]`, `sqlalchemy`, `alembic`, `pydantic-settings`, `argon2-cffi`; dev deps `ruff`, `mypy`, `pytest`, `httpx`), pinned **Python 3.13**, ruff + mypy + pytest config; commit `uv.lock`; one smoke test.
  - `frontend/`: `package.json` (pnpm; deps `react`, `react-dom`, `@mantine/core`, `@mantine/hooks`, `react-feather`, `openapi-fetch`; dev deps `vite`, `typescript`, `eslint` + config, `vitest`, `@testing-library/react`, `vite-plugin-pwa`, `openapi-typescript`), scripts `dev/build/lint/typecheck/test/codegen`; `tsconfig.json`, `vite.config.ts`, eslint + vitest config; commit `pnpm-lock.yaml`; one smoke test.
  - `ci.yml`: a **backend** job (setup uv + cache → ruff, mypy, pytest) and a **frontend** job (setup pnpm **+ store cache** + node → eslint, tsc, vitest, `vite build`). The docker + contract jobs are added in their own steps.
- **Tests:** both smoke tests pass; lint + type-check clean on both sides.
- **Done when:** push → CI green on the empty skeleton.
- **Out of scope:** no app code, models, routes, or UI.
- **Commit:** `chore: scaffold monorepo, toolchains, and CI`

### Step 2 — Backend app core (settings, app factory, health)
- **Goal:** FastAPI boots and serves health under the configurable prefix (no DB yet).
- **Build:**
  - `app/config.py`: `Settings` with `api_prefix` (default `/api`), `api_version`, `environment`, `secret_key` (**optional**, `None` by default — resolved at startup), `database_url`, `session_cookie_name`.
  - `app/main.py`: `create_app()` → FastAPI instance, mounts a root `APIRouter` under `settings.api_prefix`; no import-time side effects.
  - `app/api/routes/health.py`: `GET {api_prefix}/health` → `{status, version, api_version}` (the `db` field arrives in step 3).
- **Tests:** `TestClient` — health 200 + payload shape; settings load from env (incl. missing-secret failure).
- **Done when:** app boots via the factory; health green; gates green.
- **Out of scope:** no DB, no auth.
- **Commit:** `feat(backend): app factory, settings, and health endpoint`

### Step 3 — Persistence, migrations & data-access/context layer
- **Goal:** SQLAlchemy + Alembic wired; first migration creates the `households` singleton; all access via repository + the current-context abstraction.
- **Build:**
  - `app/db/base.py` (`DeclarativeBase`, engine, `sessionmaker`), `app/db/session.py` (`get_db` unit-of-work dependency).
  - `app/models/household.py`: `Household(id, name, currency, timezone, settings JSON)` + a **singleton invariant** (app-layer guard *and* a DB-level guard — e.g. a fixed-value unique column / single allowed PK).
  - `app/core/context.py`: `RequestContext` (current household now; current user added in step 4) + a `get_context` dependency resolving the singleton.
  - `app/repositories/household.py`: `HouseholdRepository` (`get`/`ensure` singleton) — **no raw queries in routes**.
  - Alembic: `env.py` reads `settings.database_url` + target metadata; migration `0001` creates `households` and ensures the singleton row.
  - Health gains `db: "ok"` via `SELECT 1`.
- **Tests:** `alembic upgrade head` on a fresh temp SQLite; a second-household insert is rejected; repo `get`/`ensure`; health `db:ok`.
- **Done when:** migration applies clean on an empty DB; gates green.
- **Out of scope:** no `User`/`Session` (step 4), no domain tables.
- **Commit:** `feat(backend): SQLAlchemy, Alembic, Household singleton, and context layer`

### Step 4 — Session-cookie auth skeleton
- **Goal:** login / logout / me with server-side sessions + a single bootstrapped admin.
- **Build:**
  - `app/models/user.py` (`id, email, password_hash, role, is_active, created_at`), `app/models/session.py` (`id` opaque random, `user_id`, `created_at`, `expires_at`, `last_seen_at`); migration `0002`.
  - `app/auth/passwords.py` (argon2 hash/verify), `app/auth/sessions.py` (create/verify/revoke/purge-expired).
  - `app/api/routes/auth.py`: `POST {prefix}/auth/login` (verify → create session → set `HttpOnly`+`Secure`+`SameSite=Lax` cookie), `POST {prefix}/auth/logout` (revoke + clear cookie), `GET {prefix}/auth/me` (current user / 401).
  - `app/api/deps.py`: `current_user` (cookie → validate session → user / 401); `RequestContext` now carries the user.
  - **`app_config` key/value model/repo** and **migration 0003** (reversible; creates `app_config` table).
  - **Secret key resolution** (lifespan startup): env value if set; else `app_config['secret_key']`; else generate + persist. Stash on `app.state.secret_key`.
  - **Onboarding endpoints** (unauthenticated): `GET /auth/setup-status` → `{ setup_required }` (true when zero users); `POST /auth/setup` → create first admin (409 if user already exists; no auto-login).
- **Tests (the easy-to-get-wrong logic — required):** password hash/verify; session create→verify roundtrip; **expired session rejected**; logout actually revokes (verify fails after); cookie flags present; `me` is 401 without / 200 with cookie; `app_config` get/set; secret-key resolution (env override; generate→persist→reuse; env not persisted); onboarding (`setup-status` reflects count; first setup creates hashed admin; second → 409; no auto-login); migration 0003 upgrade/downgrade clean.
- **Done when:** login sets the cookie, `me` works, logout revokes; gates green.
- **Out of scope:** no registration UI, no role enforcement beyond "is a valid user", no multi-user.
- **Commit:** `feat(backend): session-cookie auth skeleton and admin bootstrap`

### Step 5 — Contract-first codegen & no-drift CI gate
- **Goal:** generate TS types from the live OpenAPI; CI fails on drift.
- **Build:**
  - `backend/scripts/export_openapi.py`: import the app, write repo-root `openapi.json` with **stable key ordering** (deterministic diffs).
  - `Makefile` `codegen`: export `openapi.json` → `openapi-typescript openapi.json -o frontend/src/api/schema.d.ts`.
  - `frontend/src/api/client.ts`: `openapi-fetch` `createClient<paths>({ baseUrl: '', credentials: 'include' })`. (Because routers mount under `/api`, the generated `schema.d.ts` path keys already include `/api` — so `baseUrl` stays `''` and callers use the full key, e.g. `client.POST('/api/auth/login')`. Using `baseUrl: '/api'` would double-prefix to `/api/api/...`.)
  - Commit the generated `openapi.json` + `frontend/src/api/schema.d.ts`.
  - CI **contract** job: `make codegen` then `git diff --exit-code openapi.json frontend/src/api/schema.d.ts`.
- **Tests:** `make codegen` is a no-op on a clean tree (empty diff); temporarily tweak an endpoint to confirm the gate goes red, then revert — **leave nothing behind**.
- **Done when:** gate green on a clean tree, demonstrably red on drift.
- **Out of scope:** no runtime client SDK generation (types only).
- **Commit:** `build: contract-first OpenAPI→TS codegen with no-drift CI gate`

### Step 6 — Frontend shell, theme & login
- **Goal:** responsive Mantine shell + PWA + working login via the typed client.
- **Build:**
  - `src/theme.ts`: Mantine theme (primary `teal`, defaults); `MantineProvider` + color-scheme (system default + manual toggle) at the app root.
  - `src/shell/AppShell.tsx`: desktop sidebar + header; mobile burger → `Drawer`. **One** shell.
  - PWA: `vite-plugin-pwa` + web manifest + icons in `public/`; precache the app shell.
  - `src/components/`: minimal `PageShell`, `LoadingState`, `EmptyState`, `ErrorState`.
  - `src/pages/Setup.tsx`: first-run form → `client.POST('/api/auth/setup')`; on success → Login.
  - `src/pages/Login.tsx`: form → `client.POST('/api/auth/login')` → route into the shell; a logout action in the header.
  - `App.tsx` gate: `GET /api/auth/setup-status` → Setup (if required) → else `GET /api/auth/me` → shell or Login.
- **Tests (vitest + Testing Library):** Login renders + submits (client mocked); Setup renders + submits (client mocked); gate shows Setup when `setup_required:true`, Login when false+unauth, shell when authed; color-scheme toggle renders.
- **Done when:** `pnpm dev` boots; login round-trips against a running backend; installable PWA; responsive at desktop + mobile; gates green.
- **Out of scope:** no domain pages, no nav items beyond a placeholder, no component library.
- **Commit:** `feat(frontend): Mantine shell, theme, PWA, and login`

### Step 7 — Single-container Docker + prod/dev compose
- **Goal:** one image running the whole app; prod + dev compose with the override workflow.
- **Build:**
  - `docker/Dockerfile`: multi-stage — (1) `pnpm install --frozen-lockfile` + `pnpm build`; (2) `uv sync --frozen`; (3) slim final copying backend + built static; FastAPI serves `/api/*` + the SPA with **history fallback**; the **entrypoint just execs the command (no migration)**; `HEALTHCHECK` → `/api/health`; one port.
  - `docker-compose.yml` (**production**): a one-shot **`migrate`** service (`alembic upgrade head`, `restart: "no"`) + the **`app`** service from a **prebuilt `image:`** (no `build:`), SQLite on a **bind mount** (`${DATA_DIR:-./data}:/app/data`), `user: "1000:1000"`, host port `${APP_PORT:-8000}`, env from `.env`, `restart`, healthcheck. `app` `depends_on` `migrate` with `condition: service_completed_successfully` (**fail-closed**: app starts only if migration succeeds).
  - `docker-compose.dev.yml` (**dev override**): adds **`build:`** (context `.`, `docker/Dockerfile`) + dev env to both `migrate` and `app` (shared YAML anchor); stacks on top of the prod file.
  - Document the dev command: `docker compose -f docker-compose.yml -f docker-compose.dev.yml up -d --build`.
  - CI **docker** job: build the image, `docker compose config` validation, then smoke test (run migration one-shot → start app → curl `/api/health` + `/api/auth/setup-status`).
- **Tests:** image builds; `docker compose up` runs `migrate` (gated) then `app`; migrations 0001/0002/0003 apply to a fresh bind-mounted DB as uid 1000:1000; SQLite file owned by 1000:1000; `/api/health` green; setup-status reachable; **fail-closed**: a failing migration leaves `app` unstarted.
- **Done when:** prod build and dev override both run; CI docker job green.
- **Out of scope:** no k8s/orchestration, no reverse-proxy/TLS, no Postgres.
- **Commit:** `build: single-container Dockerfile and prod/dev compose`

> **Ordering rationale:** codegen (5) needs real endpoints (2, 4) to generate from; the frontend (6) needs the typed client (5); Docker/compose (7) packages the finished whole.

## 13. Post-implementation follow-ups (non-blocking carry-overs)

Recorded here (durably) when M0's milestone report was retired after the author's walkthrough. None block M1; revisit opportunistically.

1. **Test-isolation fragility (backend).** A Step-3 `importlib.reload(...)` autouse fixture mutates global state, so running some test modules **in isolation** (e.g. `pytest tests/test_step4.py` alone, or classes out of order) can error. The **full ordered CI suite is green**; this was confirmed pre-existing and non-blocking across two blind reviews. Fix opportunistically with a `chore(test)` / `conftest` pass that makes each module's settings/env isolation self-contained.
2. **`/api/health` `version` is always `"dev"`.** The backend isn't installed as a release, so `importlib.metadata.version(...)` can't resolve a real version. The contract only requires `version` to be a string (non-blocking). Wire a single version source when packaging / release versioning is added.
3. **Step-6 cosmetic observations (no rework).** `ColorSchemeScript` in `<body>` overstates FOUC protection for a CSR SPA; `useComputedColorScheme` falls back to `"dark"` (the usual convention is `"light"`); source imports omit extensions while tests use `.js` style. All cosmetic, no functional impact.

> A fourth carry-over — a design-doc `baseUrl` inconsistency — has been **resolved in place**: §9 Step 5/6 now state the implemented contract (`baseUrl: ''` + full `/api/...` keys).
